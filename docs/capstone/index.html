What Does It Do?

Abstraction

Ibis provides a layer of abstraction over Vulkan. It simplifies everything I could get it to. For example, the 1,000 lines for a triangle people talk about with Vulkan is almost entirely initialization for its structures. I packed everything that needs to be initialized into two structures: a context and renderer. They both have builders and can each be created in one line. Pipelines are even more simplified with Vulkan pipelines taking the most parameters I think anything ever will. The graphics pipeline creation function is easily over 100 lines which is way longer than I would like a function to be, but I can’t make it shorter. It’s just parameters. That can also be created in one line with a builder. Other important abstractions are buffers, which are very important to abstract because they are extremely annoying and dangerous to mangle with manually, allocators, images, and descriptor sets. Descriptor sets are how you send uniforms to shaders. I hate descriptor sets. I love everything else about the Vulkan library, but I hate descriptor sets. They are so, so confusing to me. Everything else comes to make sense once you’ve written and reread it, but not descriptor sets for me. They are also near impossible to abstract. I did recently learn about a widely supported extension that mostly replaces what I hate about them, which I will definitely consider for the future.

Optimizations

Vulkan is often said to be faster than OpenGL, and, if it’s used right, it is. However, using it right is often very hard. I tried to make this easier for the user. One of the largest optimization layers I made was an arena allocator. In Vulkan, you can only have so many VRAM allocations, and there aren’t just limited by memory space, they’re also limited by number. For example, if you wanted to make one billion allocations of one byte into VRAM, even if you had plenty of room, you couldn’t if it’s past your device limit. Because of this, Vulkan benefits greatly from arena allocators, where one allocation can be used by multiple things that need it. Your GPU also has different memory “channels,” each supporting a different memory type. For example, host visible memory that is shared between your GPU and CPU will need a different allocation than device local memory, which only the GPU sees. The arena allocator takes an array of memory types and the size of the allocation which will be shared by all descendants. This means one arena allocator can be made with enough size for your entire program and you’ll never have to worry about memory. Of course, it being an arena allocator means nothing can be freed, but you can make multiple for temporary allocations.

Parallelism

I tried to implement parallelism anywhere I could in my library since that’s another Vulkan philosophy that helps with its speed. Apart from the scary GPU synchronization barriers and semaphores, the biggest thing I went out of my way to make thread-safe was, once again, the arena allocator. Using an atomic counter, thirty threads or so could be asking for allocations at once without any trouble. At least in theory. I didn’t get to do much, but I felt it was worth noting as I will try to do more in the future.

What I Learned

I learned a lot from this project, more than any other I’ve done before. I learned everything I could about Vulkan. How to use the API, how it works, how it’s changed, even how it came to be, which is a story I like. Essentially, before Vulkan, GPU drivers and APIs had no reason to be efficient. They would bog the CPU every time they used the GPU for anything, abstracted everything too much, and were overall inefficient. AMD didn’t like poor drivers bottlenecking their hardware. To combat this, they made a low-level zero-abstraction API called Mantle. The issue with Mantle was that AMD couldn’t make it work on, say, NVIDIA GPUs, so they handed it to the people who made OpenGL to make it a cross platform standard and it became Vulkan. It then revolutionized everything and forced Windows to optimize DirectX and now even Apple is making low-level GPU APIs in Metal. It’s not really relevant to the project but I thought its origin was really cool, and it is something I learned.

Other API Workings

Funnily, I learned a lot about how other higher-level APIs worked through using Vulkan. For example, when I first made my dynamic vertex buffer, it had one frame of data that was read and written to every frame. The problem with that is that GPUs are made to be parallel and there could be millions of threads accessing the vertex data while my CPU is memcpying it, which is a very bad race condition. To combat this, I added multiple frames to the data, one per frame in flight, which is written every frame or as needed. The problem with that is now I need to know which frame in flight index we need to write to in the buffer, which I can do by having one command buffer per frame in flight (you need multiple anyway). However, when you render, the command buffer needs to know which swapchain image to render to. So to render we get all the data we need for rendering (which frame in flight index, which swapchain image), then use it to fill the command buffer with whatever we need, then write it. I came up with a system where you start the command buffer and it gives you a scene with all the data that you fill with commands, then give it back to the renderer to submit. I then realized that this is exactly what Raylib does. You call BeginRendering, tell it what to do, then call EndRendering. It even does the same thing when rendering to textures. This is more an example of me learning optimization through problem solving, but I was mind blown when I realized that whoever made Raylib probably had the exact same experience.

Vulkan Updates

One of the things I found most interesting about Vulkan was its live service. It’s constantly being updated. There’s new minor versions every few years, and they each add a lot. They add maintainance extensions which make the API (for lack of a better word) better. The also take some extensions and bake them directly into the API, meaning you no longer need the extension and the driver has to support it. I ended up using one of them, called dynamic rendering. It completely removes the need for a render pass and framebuffer, instead letting you do it all when it’s needed, and in a much cleaner way in my opinion. It means I don’t need a render pass to render to the swapchain images (what gets shown to the screen) or even a texture, which I believe would have been a huge pain otherwise. I also ended up finally understanding what a render pass was when I replaced it with dynamic rendering. I also used another baked extension called synchronization 2 which I think is a funny name but it isn’t that interesting. Keeping a Vulkan library up to date and updating it with every new big Vulkan version sounds like a lot of fun to me and I’m glad I looked into it.

What It’s Doing

First, it generates vertices with random position color and mass, then sends it to the GPU in a staged buffer. The CPU will never see the data again.

Then, every frame, a compute shader applies the gravitational force of each vertex onto the other, and moves them.

The trails are rendered to a texture, which gets darkened every frame by another compute shader.

The trails are then rendered to the texture by a graphics pipeline, using the position and velocities.

The trail texture is drawn to the screen.

The vertices are drawn pointing the direction of their velocity and with their mass as their size.

There needs to be synchronization between everything, which is done through the usage of barriers so no two things are writing to the same or reading and writing the same.

This is all done using only features the library exposes.

Learning From the Demo

Creating the demo drove me into one of the worst debugging experiences of my life. It was fun to debug on a call with my friend, but it’s the hardest bug I’ve ever had to fix, no contest.

Basically, the first time the demo “worked,” it kept doing this. I was absolutely dumbfounded because the vertices at this time were clamped to be on the screen. It should not even be possible for them to leave. Where are they going? And why the top left every single time? How do I even debug a shader?

I tried what I thought to be everything: - Maybe there’s uninitialized memory leading to a black hole of mass in the top left? Nope. I counted all 250 vertices in a screenshot. They’re all there. - Maybe they’re moving so fast you can’t see them? Nope. I clamped the velocities and they still disappear. - Maybe the y velocities are flipped or something? Nope. Trying all combinations of flipping vertices does the same thing.

I had no clue what to do other than start brute force guessing it. I can’t print from a shader, I can’t do breakpoints, I can’t even read the data every frame the CPU can’t even see it.

Eventually I realized all that would escape the clamp was a NaN value and figured out through an if statement that puts them into the center and stops them if their velocity is NaN, that their velocity was in fact, becoming NaN. This was an issue with me using a tiny area for the simulation of [-1, 1] (now [-1,000, 1,000]), which led to what I assume to be some nasty divide by near-zero math into infinity which somehow turned into NaN. I didn’t realize why they weren’t rendering until I was falling asleep that night and realized that since it uses the velocity to know which way to face, it was drawing based on trig function results of NaN, and therefore weren’t appearing since that’s garbage. They were all still there in the top left corner in the same exact spot applying insane forces to themselves and the other vertices while being completely invisible. Turns out it was an actual black hole. Maybe my gravity simulation was just too realistic. At least I know some tricks for debugging shaders now.
